EVMONITOR DASHBOARD — SECURITY OVERVIEW
========================================
Last updated: 2026-03-27


AUTHENTICATION
--------------
- Single shared password gates all access to the dashboard.
- Password is stored in the .env file on the server as DASHBOARD_PASSWORD.
  It is never hardcoded in source code and never committed to git.
- On login, the password is compared server-side using secrets.compare_digest()
  (Python stdlib). This function is timing-attack safe — it always takes the
  same amount of time regardless of how close the guess is, preventing
  attackers from measuring response time to narrow down the password.
- A 1-second delay is enforced on every failed login attempt, slowing down
  brute force attacks significantly.
- On success, the password value is used as a bearer token and stored in the
  browser's sessionStorage (not localStorage). sessionStorage is automatically
  cleared when the tab or browser is closed, so sessions do not persist.


API PROTECTION
--------------
- Every API endpoint (/api/*) is protected by HTTP middleware that checks for
  a valid Authorization: Bearer <token> header on every request.
- The login endpoint (/api/auth) is the only route exempt from this check.
- Direct API calls without a valid token (e.g. via curl or a script) receive
  a 401 Unauthorized response and no data.
- This means the protection is at the data layer, not just the UI layer.
  Bypassing the frontend does not grant access to the data.
- Port 8000 (uvicorn) is bound to 127.0.0.1 only. It is not reachable from
  the internet — all traffic must go through nginx.


TRANSPORT SECURITY (HTTPS)
--------------------------
- nginx is configured as a reverse proxy terminating TLS on port 443.
- A self-signed certificate is used (generated with openssl, valid 10 years).
  The certificate includes a Subject Alternative Name (SAN) for the server IP,
  which is required by modern browsers.
- HTTP (port 80) permanently redirects to HTTPS (301).
- The following security headers are set by nginx:
    Strict-Transport-Security  — instructs browsers to always use HTTPS
    X-Frame-Options            — prevents clickjacking (SAMEORIGIN)
    X-Content-Type-Options     — prevents MIME-type sniffing
    Referrer-Policy            — no referrer header sent on navigation

SELF-SIGNED CERT NOTE:
  The certificate is not issued by a trusted Certificate Authority, so browsers
  will show a "Your connection is not private" warning on first visit. The
  traffic is still fully encrypted. Click "Advanced → Proceed" to continue.
  To eliminate the warning permanently, install the cert as a trusted root in
  your OS/browser, or switch to a domain-based Let's Encrypt certificate.

DOMAIN UPGRADE PATH:
  If a domain name is later obtained (or a free subdomain via duckdns.org):
  1. Switch from nginx to Caddy in docker-compose.yml
  2. Configure the Caddyfile with the domain — Caddy auto-fetches Let's Encrypt
  3. Update CORS_ORIGINS in .env to https://yourdomain.com
  4. The browser warning disappears permanently; cert renews automatically


FRONTEND
--------
- The React app checks sessionStorage on load. If no valid token is present,
  the login page is shown before any part of the dashboard renders.
- All API calls from the frontend use an authFetch() wrapper that
  automatically injects the Authorization header on every request.
- This covers all data-fetching components: Live Status, Analytics, Map,
  Hub Detail modal, Custom Range panels, and the Excel export.
- The static HTML/CSS/JS files (the app shell) are served without auth.
  This is intentional and safe — no sensitive data lives in the frontend
  bundle itself.


KNOWN LIMITATIONS
-----------------
- Self-signed certificate: Browsers warn on first visit (see above). Traffic
  is still encrypted. A trusted cert requires a domain name.

- Single shared password: There are no individual user accounts, no roles,
  and no audit log of who accessed what. If the password is compromised,
  change it in .env and restart the API container.

- No token expiry: The bearer token does not expire server-side. It remains
  valid until DASHBOARD_PASSWORD is changed in .env.

- Rate limiting: The 1-second delay on failed logins is a basic deterrent,
  not a full rate limiter. A determined attacker could make ~60 attempts/min.
  Stronger rate limiting could be added at the nginx layer (limit_req_zone).

- CORS: The API should have CORS_ORIGINS set to your server IP or domain in
  .env on the server. The default (*) allows any origin to make API requests
  if they obtain your token.


RECOMMENDATIONS
---------------
1. Set DASHBOARD_PASSWORD in .env to a strong, random passphrase.
2. Set CORS_ORIGINS=https://YOUR_SERVER_IP in .env on the server, then
   restart the API: docker compose restart api
3. To eliminate browser cert warnings: obtain a domain name and switch to
   Caddy + Let's Encrypt (see Domain Upgrade Path above).
4. Optional: add a Hetzner Cloud Firewall in the Hetzner console to allow
   only ports 80 and 443 inbound, as a defence-in-depth layer.
